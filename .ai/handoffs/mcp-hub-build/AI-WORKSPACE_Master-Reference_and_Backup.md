# سند مرجع و بک‌آپ کامل پروژه AI-WORKSPACE
### منبعِ حقیقت (Source of Truth) + بک‌آپ دستاوردها + فلسفه‌ی درست بانک MCP + نقشه‌ی راه

> این سند دو کار می‌کند: (۱) همه‌ی دستاوردها و تصمیم‌های پروژه را یک‌جا نگه می‌دارد تا هیچ‌چیز گم نشود، (۲) رؤیای کامل و تصحیح‌شده‌ی «بانک MCP مرکزیِ همیشه‌در‌حال‌یادگیری» را ثبت می‌کند. سند اجراییِ فاز اول جداست (فایل «Handoff for Roo Code»)؛ این سند بالادستیِ آن است.

## بخش ۰ — پروژه در یک نگاه

AI-WORKSPACE یک «سیستم‌عاملِ کوچک برای هوش مصنوعی» است؛ یک محیط کارِ مبتنی بر عامل (Agentic Workspace) که در آن دانش، قوانین، مهارت‌ها و ابزارها **یک بار با دقت ساخته می‌شوند** و در همه‌ی پروژه‌های بعدی دوباره استفاده می‌شوند.

**اصل بنیادی: «سرمایه‌ی دائمی».** پروژه‌ی اول کند است چون پایه ساخته می‌شود؛ از پروژه‌ی چهارم به بعد، ساختِ یک پروژه‌ی جدید تقریباً فقط با «تعریف هدف» شروع می‌شود، چون Agentها، Rules، Skills، Templates و **ابزارهای MCP** از قبل آماده‌اند.

**هدف نهایی (سه لایه):**
1. سیستمی که پروژه‌های نرم‌افزاری‌ات را با کیفیت حرفه‌ای می‌سازد.
2. سیستمی که **همیشه در حال یادگیری** است: هر کارِ جدید، ابزارهای لازمش را می‌سازد؛ کارهای بعدی از همان ابزارها استفاده می‌کنند و فقط برای بخش‌های تازه ابزار جدید ساخته می‌شود.
3. سیستمی که از راه دور (تلگرام) کنترل می‌شود تا حتی وسط پیاده‌روی هم بتوانی پروژه بسازی و مدیریت کنی.

## بخش ۱ — بک‌آپ دستاوردها و تصمیم‌ها (هیچ‌کدام حذف نشود)

### ۱.۱. زیرساختِ ساخته‌شده (وضعیت فعلی)
- پوشه‌ی مرکزی `AI-WORKSPACE` با ساختار حرفه‌ای (لایه‌های L0–L5، `.agents`، `rules`، `state`).
- هسته‌ی پردازشی: **Roo Code** در محیط پایدار (Antigravity) نصب و به مدل استدلالی متصل است.
- **ورودی صوتی (Whisper)** برای Voice-Prompt فعال است.
- **دسترسی فایل** تأیید شده (خواندن/نوشتن لوکال).
- **اتصال فعلی به Jira برقرار و کارکننده است** ← حذف نمی‌شود؛ نقشش «provider پشتیبان (fallback)» می‌شود. (روش دقیقش با پرامپت Codex در پیوست الف کشف می‌شود.)
- **Obsidian** به‌شکل حرفه‌ای راه‌اندازی شده ← لایه‌ی حافظه‌ی انسان‌خوان.

### ۱.۲. تصمیم‌های معماری تثبیت‌شده (از Master Plan v2/v3 — پابرجا)
هسته:
- **.NET Clean Architecture / Modular Monolith** (نه میکروسرویس از روز اول).
- **Python FastAPI** برای سرویس‌های AI (مغز AI جدا، قابل‌تست، قابل‌deploy).
- **PostgreSQL + EF Core**؛ برای RAG اولیه **pgvector**، و **Qdrant** فقط وقتی workload سنگین شد.
- ارتباط .NET و Python: در شروع REST/JSON؛ بعداً برای کارهای طولانی queue (مثل RabbitMQ).
- **Docker Compose روی VPS** برای دیپلوی اول؛ Kubernetes/Helm فقط future option.
- **GitHub Actions** موتور اصلی CI/CD.
- **DevOps + LLMOps از روز اول**.
- **OpenTelemetry از روز اول** (trace/metrics/logs با correlation ID بین .NET، Python، RAG، LLM).
- **.NET Aspire** برای orchestration لوکال و dashboard توسعه‌دهنده.
- **Renovate/Dependabot** برای آپدیت کنترل‌شده‌ی dependency با PR.
- **OpenTofu یا Pulumi** برای IaC بعد از MVP (یکی، نه هر دو).
- **MCP-friendly design** پشت Gateway/Broker امن ← (موضوع اصلی این پروژه، بخش ۲).
- **Semantic Kernel / Microsoft Agent Framework** به‌عنوان لایه‌ی agent آینده وقتی workflowها پیچیده شدند.
- Future options شناخته‌شده اما فعلاً نه: **Dapr، Temporal**.

Schemaهای دیتابیس (تثبیت‌شده): `app, identity, jobs, documents, ai, llmops, feedback, eval, audit, outbox`.

LLMOps از روز اول: Prompt Registry و نسخه‌بندی Prompt، Model Versioning و Release Channels، RAG Index Versioning، Evaluation Pipeline، Feedback Store، Trace Store.

پروژه‌ی نمونه‌ی اول (که در اسناد آمده): «سیستم اتوماسیون درخواست شغل با CV سفارشی و امتیازدهی AI».

### ۱.۳. چارچوب نظری (پابرجا و ارزشمند)
- **بلوغِ Loop Engineering:** Prompt → Context → Harness → **Loop Engineering** (تصمیم‌گیرنده داخل حلقه است). مدل ذهنی دقیق‌تر: «ساختنِ سیستم با عامل‌ها» با مرز ورودی=Open Task و خروجی=Verified Pull Request.
- **معماری چندلایه L0–L5:** L0 Universal / L1 Domain / L2 Stack / L3 Blueprint / L4 Task / L5 Learning.
- **مؤلفه‌های حلقه:** Automations (تریگرها)، Work Trees (محیط‌های موازی)، Skills (یک‌کار‌یک‌فایل)، Plugins/Connectors (که MCP استانداردِ آن است)، Sub-Agents (جداکردنِ سازنده از ممیز)، Memory (حالتِ ماندگار در repo).
- **Hermes (خودبهبودده):** Telegram Gateway، فایل‌های حافظه‌ی سخت‌گیرانه (USER.md، MEMORY.md با محدودیت کاراکتر برای جلوگیری از تورم context)، **Skill Crystallization** (بعد از هر موفقیت، منطق تکرارشونده به یک فایل ماژولار .py تبدیل می‌شود)، **Curator** (چرخاندن مهارت‌ها بین Active/Stale/Archive تا از «انبوهگیِ ابزار» و خطای انتخاب ابزار جلوگیری شود)، «10-turn nudge» برای به‌روزرسانی حافظه.
- **اکوسیستم Ruflo (مرجع مفهومی):** ruflo-core (کشف پلاگین)، autopilot (اجرای خودمختار حلقه)، agentdb (وکتور HNSW)، ruvector/rag-memory (جستجوی هیبرید و Graph RAG)، intelligence (خودیادگیری با الگوهای SONA)، goals (تجزیه‌ی هدف با GOAP)، graph-intel.
- **امنیت:** AIDefence (بلاک prompt injection)، Witness Verification (اثبات رمزنگاشتیِ تطابق بایت‌های نصب‌شده با manifest امضاشده)، مدل Zero-Trust Federation با امتیاز اعتماد.

### ۱.۴. قوانین غیرقابل مذاکره (پابرجا)
- main همیشه پایدار؛ **push مستقیم به main ممنوع**؛ هر تغییر از مسیر **Pull Request**؛ هر PR باید checkهای خودکار را پاس کند؛ حل هر conflict کل pipeline مربوطه را دوباره تریگر می‌کند.
- **Self-learning مستقیم ممنوع:** هیچ feedback خامی مستقیم وارد prompt/RAG/model/memoryِ production نمی‌شود. مسیر درست: feedback خام → review/approval → offline evaluation → candidate version → staging → canary → فقط اگر metrics بهتر شد، production.
- تغییر معماری فقط با تأیید Architect و ثبت **ADR**.
- هیچ Secret داخل repository ذخیره نمی‌شود.

## بخش ۲ — فلسفه‌ی درستِ بانک MCP مرکزی (مدل ذهنی تصحیح‌شده)

### ۲.۱. MCP واقعاً چیست
MCP یک استاندارد بازِ Anthropic (نوامبر ۲۰۲۴؛ در دسامبر ۲۰۲۵ به بنیاد Agentic AI زیر Linux Foundation اهدا شد) برای وصل‌کردن برنامه‌های AI به سیستم‌های بیرونی است. تشبیه رسمی: **«USB-C برای هوش مصنوعی»**. سه نقش دارد:
- **Host (میزبان):** خودِ برنامه/عاملِ AI که تصمیم می‌گیرد. (اینجا: Orchestrator تو.)
- **Client (کلاینت):** مترجمِ داخل میزبان که با سرور حرف می‌زند.
- **Server (سرور):** سرویسی که قابلیت‌ها را عرضه می‌کند.

هر سرور سه چیز عرضه می‌کند:
- **Tools (ابزار):** کارهایی که *اثر* دارند (فرستادن ایمیل، کوئری زدن). مدل تصمیم می‌گیرد کِی صداشان بزند.
- **Resources (منابع):** داده‌ی فقط‌خواندنی برای context (بدون side effect).
- **Prompts:** قالب‌های workflow قابل‌استفاده‌ی مجدد.

### ۲.۲. نکته‌ی طلایی (تصحیح مدل ذهنی)
> **سرور MCP یک «جعبه‌ابزار» است، نه یک عاملِ مستقلِ خودگردان.** تصمیم‌گیرنده در Host/عامل می‌ماند؛ سرور فقط ابزار عرضه می‌کند و ابزارها را مدل صدا می‌زند.

یعنی «MCP ایمیل را صدا بزن» دقیق‌ترش این است: یک **سرور ایمیل** داری با چند ابزار مرتبط (خواندن، فرستادن، جستجو، برچسب)، و **عاملِ تو** ترتیب و انتخابشان را مدیریت می‌کند. کنترل، ممیزی و امنیت دست تو می‌ماند.

### ۲.۳. دو قاعده‌ی مهم که از تحقیق روز درآمد
1. **برای هر کار کوچک یک سرور جدا نساز.** یک سرور می‌تواند چند ابزار مرتبط را با هم داشته باشد (مثل سرور GitHub که هم‌زمان جستجوی مخزن، ساخت issue و مدیریت PR را دارد). اگر صدها ابزار پراکنده بسازی، انتخابِ ابزارِ مدل خراب می‌شود. راهکار استاندارد: گروه‌بندی ابزارها در سرورها + فعال‌کردن فقط زیرمجموعه‌ی لازم برای هر پروژه (**Tool Groups**). این دقیقاً همان نقشِ **Curator** در Hermes است.
2. **الگوی Gateway/Registry استاندارد صنعتی است.** «همه‌ی سرورها یک‌بار ثبت شوند، همه‌ی کلاینت‌ها (Claude/Cursor/Codex/عامل‌های خودت) به یک نقطه‌ی واحد وصل شوند.» این همان تصمیمِ Broker/Gateway توست. نمونه‌های بالغ: MCPJungle، Docker MCP Gateway، MetaMCP. ویژگی‌های کلیدی‌شان: RBAC، audit log، مدیریت متمرکز secret، مشاهده‌پذیری با OpenTelemetry.

### ۲.۴. درسِ امنیتی (که تصمیم قبلی‌ات را تأیید می‌کند)
هزاران سرور MCP بدون احراز هویت در اینترنت رها شده‌اند؛ حمله‌هایی مثل tool poisoning و over-permissioning واقعی‌اند؛ یک ایجنت Replit با وجود «فریز» یک دیتابیس production را پاک کرد. **نتیجه:** طراحی human-in-the-loop و مسیر اجباریِ Gateway → permission → validation → execution → audit، یک تجملِ اختیاری نیست؛ خطِ قرمزِ ایمنی است.

### ۲.۵. خودیادگیریِ درست («اول MCP را بنویس، بعد استفاده کن») — با قید
- منطق: وقتی عامل برای کاری ابزاری در catalog ندارد → workflow `create-new-mcp-server` فعال می‌شود → از روی template یک سرور می‌سازد → پشت Gateway ثبت می‌شود → تست می‌شود → استفاده می‌شود.
- **قید حیاتی:** این کار **gated** است. در نقاط حساس (شبکه، دیتابیس production، secret، عملیات مخرب) **تأیید انسان اجباری است**. این با قانون «Self-learning مستقیم ممنوع» و درس Replit سازگار است.
- **کِی «crystallize» کنیم:** نه برای هر کار یک‌بارمصرف. وقتی یک الگو **تکرارشونده** شد، آن را به یک سرور/skill ماندگار تبدیل کن (Skill Crystallization در Hermes). این‌طور بانک بدون تورم رشد می‌کند: هدف این است که به مرور مثلاً ۸۰٪ یک پروژه از ابزارهای موجود ساخته شود و فقط برای ۲۰٪ باقی، ابزار تازه نوشته شود.

## بخش ۳ — جای Obsidian و تلگرام در معماری

هر دو، لایه‌هایی روی همین سیستم‌اند، نه چیز موازی:

- **Obsidian = لایه‌ی حافظه‌ی انسان‌خوان.** `.ai/10-memory-and-lessons/` را ماشین می‌نویسد؛ Obsidian همان دانش را برای تو قابل‌مرور و قابل‌ربط‌دادن می‌کند. گام حرفه‌ای بعدی: یک **سرور MCP برای Obsidian** تا عامل‌ها مستقیم در vault تو یادداشت بخوانند/بنویسند و حافظه‌ی تو و سیستم یکی شود.
- **تلگرام = لایه‌ی کنترل از راه دور (Telegram Gateway در Hermes).** وقتی هسته کامل شد، تلگرام یک کلاینت/تریگر می‌شود: از روی گوشی به Orchestrator فرمان می‌دهی، او کار را بین عامل‌ها پخش می‌کند و گزارش را به تلگرامت برمی‌گرداند. مدیریت ساختِ پروژه حتی وسط پیاده‌روی.

## پیوست الف — پرامپت آماده برای Codex (کشف اتصال فعلی Jira)

> این پرامپت فقط «تحقیق و گزارش» است؛ Codex نباید چیزی را تغییر دهد.

```text
نقش تو: بازرسِ فقط-خواندنی. هیچ فایلی را تغییر نده، هیچ چیزی نصب یا حذف نکن،
هیچ commit ای نزن. فقط بررسی کن و گزارش بده.

هدف: می‌خواهم دقیق بفهمم این پروژه/محیط الان «چطور» به Jira وصل شده است.

این موارد را بررسی و در گزارشت مشخص کن:
1. روش اتصال چیست؟ (کدام یک: connector آماده‌ی Atlassian/MCP موجود،
   یک اسکریپت با Jira REST API، یک SDK، Webhook، یا روش دیگر؟)
2. فایل‌ها و مسیرهای دخیل را فهرست کن (اسکریپت‌ها، config، env).
3. Endpoint/Base URL و نسخه‌ی API که استفاده شده چیست؟ (مثلاً /rest/api/3)
4. نوع احراز هویت چیست؟ (API Token، OAuth، Basic Auth، PAT؟)
   نام متغیرهای محیطی مربوطه را بگو، اما **هرگز مقدار واقعی secret را چاپ نکن**.
5. دقیقاً چه کارهایی روی Jira انجام می‌شود؟ (خواندن issue، ساخت issue،
   تغییر وضعیت، کامنت، جستجو با JQL و...)
6. کدام کتابخانه‌ها/dependency ها برای این اتصال استفاده شده‌اند؟
7. اگر تست یا نمونه‌ی فراخوانی وجود دارد، مسیرش را بگو.

قالب خروجی: یک خلاصه‌ی مرتب به این شکل بده:
- روش اتصال:
- فایل‌های دخیل:
- Base URL و نسخه API:
- نوع احراز هویت + نام env variableها (بدون مقدار):
- عملیات پشتیبانی‌شده روی Jira:
- dependency ها:
- نکات/ریسک‌هایی که دیدی:

در پایان بگو: «برای تبدیل این روش به یک provider پشتیبان (fallback) در کنار یک
سرور MCP جدید، چه چیزهایی باید ثابت بماند و چه چیزهایی باید انتزاعی (abstract) شود؟»
```
