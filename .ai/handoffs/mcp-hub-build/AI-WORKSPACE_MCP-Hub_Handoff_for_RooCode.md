# منشور پروژه و راهنمای ادامه‌کار برای Roo Code
### پروژه: AI-WORKSPACE — سیستم چندلایه‌ی ایجنتی و بانک مرکزی MCP

> این سند برای Roo Code نوشته شده تا **بدون از دست دادن هیچ‌کدام از تصمیم‌ها و کارهای قبلی**، بانک MCP را به‌درستی داخل معماری موجود جا بیندازد و از آن‌جا کار را ادامه دهد. لطفاً قبل از هر اقدام، کل این سند را بخوان و سپس طبق بخش «اولین کارهای عملی» جلو برو.

---

## ۰. این سند را چرا می‌خوانی؟

تو (Roo Code) قرار است هسته‌ی اجرایی این سیستم باشی. پیش از این، معماری کامل پروژه در چند سند طراحی شده (سیستم چندلایه‌ی ایجنتی، Master Plan نسخه‌ی ۲ و ۳، و Blueprintهای Loop Engineering). حالا یک تصمیم جدید گرفته شده:

**بانک MCP به‌عنوان یک لایه‌ی مرکزی مشترک وارد سیستم می‌شود، نه به‌عنوان یک پروژه‌ی جدا و نه زیر یک پروژه‌ی خاص.**

وظیفه‌ی تو پیاده‌سازی همین تصمیم است، با رعایت قوانینی که پایین آمده.

---

## ۱. پروژه چیست و هدف نهایی چیست؟

AI-WORKSPACE یک «سیستم‌عامل کوچک برای هوش مصنوعی» است. هدف این نیست که از AI مثل یک چت‌بات استفاده شود؛ هدف ساختن یک **محیط کار مبتنی بر عامل (Agentic Workspace)** است که در آن دانش، قوانین، مهارت‌ها و ابزارها **یک بار با دقت ساخته می‌شوند** و در همه‌ی پروژه‌های بعدی دوباره استفاده می‌شوند.

**اصل بنیادی:** «سرمایه‌ی دائمی». پروژه‌ی اول زمان بیشتری می‌گیرد چون پایه ساخته می‌شود؛ از پروژه‌ی چهارم به بعد، ساختن یک پروژه‌ی جدید فقط با تعریف هدف آن شروع می‌شود، چون Rules، Skills، Agents، Templates و حالا **ابزارهای MCP** از قبل آماده‌اند.

**نتیجه‌ی مورد انتظار:** کیفیت کار دیگر به حافظه‌ی لحظه‌ای هیچ مدلی (Gemini، Claude، Codex و...) وابسته نیست؛ به فایل‌های مستند، نسخه‌بندی‌شده و قابل‌ممیزی وابسته است.

## ۲. وضعیت فعلی (چه چیزهایی ساخته شده — اینها را حفظ کن)

- **پایگاه مرکزی ساخته شده:** پوشه‌ی `AI-WORKSPACE` با ساختار حرفه‌ای (`.ai/`، لایه‌های L0–L5، rules، agents، state).
- **هسته‌ی پردازشی مستقر شده:** خودِ Roo Code در محیط پایدار نصب و به مدل استدلالی متصل است.
- **دسترسی فایل تأیید شده:** سیستم می‌تواند فایل‌های لوکال را بخواند و در آن‌ها بنویسد.
- **ورودی صوتی فعال است:** Whisper برای Voice-Prompt راه‌اندازی شده.
- **اتصال فعلی به Jira برقرار است:** یک روش کاری برای وصل شدن به Jira **همین الان کار می‌کند**. این روش نباید حذف شود؛ نقش آن در سیستم جدید تبدیل می‌شود به **provider پشتیبان (fallback)**.
- **تصمیم‌های معماری تثبیت شده‌اند:** .NET Clean / Modular Monolith + Python FastAPI برای AI + PostgreSQL/EF Core + Docker Compose + GitHub Actions + LLMOps/DevOps از روز اول. و مهم‌تر از همه برای این سند: **«MCP-friendly design پشت یک Gateway/Broker»** به‌عنوان تصمیم رسمی نسخه‌ی ۳ ثبت شده است.

> نکته‌ی کلیدی: بانک MCP یک ایده‌ی بیرونی و تازه نیست. این سند دارد جای خالی‌ای را پر می‌کند که معماری از قبل برایش صندلی گذاشته بود.

## ۴. تصمیم جدید: بانک MCP دقیقاً کجا و چگونه

### ۴.۱. اصل تصمیم
بانک MCP یک لایه‌ی مرکزی مشترک است: `‎.ai/11-mcp-hub/`. چون سرورهای MCP «ابزارِ قابل‌استفاده‌ی مجدد بین پروژه‌ها» هستند، دقیقاً از جنس همان سرمایه‌ی دائمی‌اند و باید کنار بقیه‌ی لایه‌های مشترک بنشینند، نه داخل یک پروژه.

### ۴.۲. ساختار پوشه‌ی بانک MCP
```text
.ai/11-mcp-hub/
├── README.md                       # توضیح بانک + نحوه‌ی استفاده
├── registry/
│   ├── catalog.md                  # کاتالوگ همه‌ی سرورهای MCP + وضعیت هرکدام
│   └── capabilities.md             # نگاشت «قابلیت → provider(ها)»  ← قلب الگوی Fallback
├── gateway/
│   ├── broker-config.md            # پیکربندی Broker/Gateway مرکزی
│   ├── permissions.md              # سیاست دسترسی هر سرور (چه ایجنتی، چه ابزاری)
│   └── audit-policy.md             # لاگ، rate limit، و ثبت هر فراخوانی
├── servers/
│   ├── system/                     # ابزارهای سیستمی
│   │   ├── filesystem/
│   │   └── db-sqlite-postgres/
│   ├── cloud/                      # ابزارهای ابری/سازمانی
│   │   ├── github/
│   │   └── jira/                   # ★ اولین سرور اختصاصی ما
│   │       ├── server/             # کد سرور MCP جدید Jira (provider اصلی)
│   │       ├── manifest.md         # نام، ابزارها، schema ورودی/خروجی، permissionها
│   │       └── fallback.md         # اشاره به روش فعلی Jira به‌عنوان provider پشتیبان
│   └── custom/                     # سرورهای شخصی که خودت بعداً می‌نویسی
├── templates/
│   ├── mcp-server-python-template/ # قالب آماده برای ساخت سرور جدید با Python
│   └── mcp-server-node-template/   # قالب آماده برای ساخت سرور جدید با Node
└── config/
    └── mcp_settings.reference.json # مرجعِ نسخه‌بندی‌شده که با mcp_settings.json خود Roo Code سینک می‌شود
```

### ۴.۳. قانون طلایی (از تصمیم رسمی نسخه‌ی ۳ — غیرقابل مذاکره)
هیچ سرور MCP نباید مستقیم و آزاد به سیستم دسترسی داشته باشد. مسیر هر فراخوانی باید این باشد:

```text
Agent / LLM
  → MCP Client
  → Tool Gateway / MCP Broker
  → Permission check          (این ایجنت اجازه‌ی این ابزار را دارد؟)
  → Input schema validation   (ورودی معتبر است؟)
  → Tool execution
  → Output validation
  → Audit log                 (چه کسی، چه زمانی، چه چیزی)
  → Response to agent
```

## ۵. الگوی Fallback برای Jira (و هر ابزار دیگری در آینده)

این همان چیزی است که می‌خواهی: **سیستم فعلی Jira بماند، یک MCP جدید هم بنویسیم، و اگر MCP کار نکرد خودکار به سیستم قبلی سوییچ شود.**

راهش این است که ایجنت هیچ‌وقت مستقیم به «Jira» وصل نشود؛ به یک **قابلیت** وصل شود. در `registry/capabilities.md` این‌طور تعریف می‌شود (به‌صورت مفهومی):

```yaml
capability: issue-tracking
  policy: از provider با بالاترین اولویتِ سالم استفاده کن؛ اگر health_check رد شد، به اولویت بعدی سوییچ کن و لاگ بزن.
  providers:
    - id: jira-mcp          # سرور MCP جدید
      type: mcp
      priority: 1           # اصلی
      health_check: ping ابزار Jira-MCP از طریق Broker
    - id: jira-legacy       # روشِ فعلی که همین الان کار می‌کند
      type: existing        # دست‌نخورده باقی می‌ماند
      priority: 2           # پشتیبان (fallback)
      health_check: تست اتصال روش فعلی
```

با این کار: چیزی پاک نمی‌شود، MCP جدید primary است، و در صورت خرابی، بازگشت خودکار به روش قدیمی تضمین می‌شود. **همین الگو دقیقاً برای GitHub، دیتابیس و هر سرویس بعدی هم تکرار می‌شود.**

## ۶. قابلیت «سیستم اول اسکریپت MCP خودش را بنویسد، بعد استفاده کند»

این را به‌صورت یک workflow و skill کنترل‌شده اضافه می‌کنیم — نه دسترسی آزاد. منطق:

1. Orchestrator تشخیص می‌دهد برای انجام کار، ابزاری لازم است که در `registry/catalog.md` موجود نیست.
2. Workflow با نام `create-new-mcp-server` فعال می‌شود.
3. ایجنت از روی `templates/` یک سرور MCP جدید scaffold می‌کند (Python یا Node).
4. سرور جدید حتماً پشت Gateway/Broker ثبت می‌شود و `manifest.md` و permission آن نوشته می‌شود.
5. در نقاط حساس (دسترسی به شبکه، دیتابیس production، secretها) **تأیید انسان (human-in-the-loop) اجباری است**.
6. بعد از تست، سرور به `catalog.md` اضافه و تازه آن‌وقت استفاده می‌شود.
7. اگر ابزار قابل‌استفاده‌ی مجدد بود، یک درس‌آموخته در `10-memory-and-lessons/` ثبت می‌شود.

> این دقیقاً با قوانین موجود سازگار است: «MCP از روز اول به‌معنی دسترسی آزاد AI نیست» و «Self-learning کنترل‌نشده ممنوع».

## ۷. آینده: چطور همه‌ی پروژه‌ها به‌صورت repo زیر این قرار می‌گیرند

هر پروژه‌ی جدید یک **repository مستقل** کنار `.ai/` است، ولی از سیستم مرکزی و بانک MCP مشترک تغذیه می‌کند:

```text
my-new-project/
├── .project-ai/
│   ├── project-brief.md        # کاربر فقط هدف پروژه را این‌جا می‌نویسد
│   ├── architecture.md
│   ├── agent-routing.md
│   ├── selected-skills.md
│   ├── selected-rules.md
│   ├── selected-mcp-servers.md # ★ کدام ابزارهای بانک MCP در این پروژه مجازند
│   ├── quality-gates.md
│   └── decisions/ADR-0001-*.md
├── src/  ├── tests/  ├── docs/
├── docker-compose.yml
└── README.md
```

- قوانین **عمومی** از `workspace-root/.ai/` خوانده می‌شوند.
- قوانین **خاصِ پروژه** در `.project-ai/` می‌مانند.
- ابزارها از **بانک مرکزی MCP** برداشته می‌شوند؛ فقط آن‌هایی که در `selected-mcp-servers.md` مجاز شده‌اند در آن پروژه فعال‌اند (اصل کمترین دسترسی).

**جریان ساخت پروژه‌ی جدید:** کاربر هدف را در `project-brief.md` می‌نویسد → Orchestrator نوع پروژه را تشخیص می‌دهد → Rules/Skills/MCP-Serverهای مناسب انتخاب می‌شوند → Architect معماری را می‌چیند → ایجنت‌های تخصصی کار + تست را انجام می‌دهند → Reviewer با Quality Gates می‌سنجد → Merge → درس‌آموخته ثبت می‌شود.

## ۸. قوانین غیرقابل مذاکره‌ی مرتبط با MCP

1. هیچ سرور MCP بدون عبور از Gateway/Broker فراخوانی نمی‌شود.
2. هیچ Secret داخل repository یا داخل کد سرور MCP ذخیره نمی‌شود.
3. هر سرور MCP باید `manifest.md` (نام، ابزارها، schema، permission) داشته باشد.
4. تغییر معماری فقط با تأیید Architect و ثبت ADR مجاز است.
5. هر تغییر در Branch جدا و از طریق Pull Request؛ push مستقیم به main ممنوع.
6. ساخت خودکار سرور MCP در نقاط حساس نیازمند تأیید انسان است.
7. روش فعلی Jira حذف نمی‌شود؛ فقط به provider پشتیبان تبدیل می‌شود.

## ۹. اولین کارهای عملی که باید انجام دهی (به‌ترتیب)

1. **Scaffold ساختار بانک:** پوشه‌ی `.ai/11-mcp-hub/` را با تمام زیرپوشه‌های بخش ۴.۲ بساز و در هرکدام یک `README.md` کوتاه توضیحی قرار بده.
2. **ساخت Gatewayِ مفهومی:** فایل‌های `gateway/broker-config.md`، `permissions.md` و `audit-policy.md` را طبق «قانون طلایی» (بخش ۴.۳) بنویس.
3. **ساخت رجیستری قابلیت‌ها:** `registry/capabilities.md` و `registry/catalog.md` را بساز و اولین قابلیت `issue-tracking` را با دو provider (jira-mcp و jira-legacy) طبق بخش ۵ ثبت کن.
4. **ساخت Templateها:** حداقل `templates/mcp-server-python-template/` را با یک سرور MCP حداقلی و قابل‌اجرا آماده کن تا مبنای همه‌ی سرورهای بعدی باشد.
5. **اولین سرور واقعی — Jira:** در `servers/cloud/jira/` سرور MCP جدید Jira را از روی template بساز، `manifest.md` و `fallback.md` را بنویس، و آن را در `catalog.md` ثبت کن. روش فعلی Jira را دست نزن.
6. **سینک تنظیمات:** `config/mcp_settings.reference.json` را بساز و آن را با `mcp_settings.json` خودِ Roo Code هماهنگ کن، سپس سیستم را ری‌استارت کن تا ابزار در پنل ظاهر شود.
7. **تست Fallback:** یک بار با MCP سالم و یک بار با MCP عمداً خاموش تست کن که سوییچ به provider قدیمی درست کار می‌کند و لاگ ثبت می‌شود.
